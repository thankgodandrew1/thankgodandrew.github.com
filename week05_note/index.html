3.1 Fetch:

The Fetch API provides an interface for fetching resources. It can be used to fetch any type of resource, including images, videos, and JSON data, and all that without reloading the page.

The basic syntax of fetch() is:

  let promise = fetch(url, [options]) 

  url - the URL to access.
  options - optional parameters: 
            method, headers etc.

Without options, this is a simple GET request, downloading the contents of the url. Getting a response is usually a two-stage process.

  1. First, the promise, returned by fetch, resolves with an object of the built-in Response class as soon as the server responds with headers.
    At this stage we can check HTTP status, to see whether it is successful or not, check headers, but don't have the body yet.
    The promise rejects if the fetch was unable to make HTTP-request, e.g. network problems, or there's no such site. Abnormal HTTP-statuses, such as 404 or 500 do not cause an error.
  2. Second, to get the response body, we need to use an additional method call.

Response provides multiple promise-based methods to access the body in various formats:

  • response.text() - read the response and return as text,
  • response.json() - parse the response as JSON,
  • response.formData() - return the response as FormData object
  • response.blob() - return the response as Blob (binary data with type),
  • response.arrayBuffer() - return the response as ArrayBuffer (low-level representation of binary data),
  • additionally, response.body is a ReadableStream object, it allows you to read the body chunk-by-chunk.

There are also different fetch options which includes: method, headers, and body

3.5 Fetch: Cross-Origin Requests

Fetch is a modern JavaScript API for making HTTP requests. By default, fetch() does not allow for cross-origin requests. This means that a request made from one website to a different website (i.e. a different origin) will be blocked.

However, it is possible to perform cross-origin requests with fetch() by using the following two methods:

  1. Setting the "mode" option in the fetch() request to "cors"
  2. Providing an "Access-Control-Allow-Origin" header in the response from the server

Example usage of fetch() with "mode" option set to "cors":

  fetch('https://example.com', {
    mode: 'cors'
  })
  .then(response => {
    if (!response.ok) {
      throw new Error(`Network
       response was not ok`);
    }
    return response.json();
  })
  .then(data => {
    console.log(data);
  })
  .catch(error => {
    console.error(`There was a 
    problem with the 
    fetch operation:`, error);
  });

Note: Cross-origin requests made with fetch() are subject to the same-origin policy, unless the server explicitly allows cross-origin requests by providing the appropriate headers.

What is Preflight Request?

A CORS preflight request is a CORS request that checks to see if the CORS protocol is understood and a server is aware using specific methods and headers. It is an OPTIONS request, using three HTTP request headers: Access-Control-Request-Method , Access-Control-Request-Headers , and the Origin header.

Single Page Application (SPA)

      

        
A single-page application (SPA) is a web application or website that
          interacts with the user by dynamically rewriting the current web page
          with new data from the server, instead of loading entire new pages
          from a server. This gives the user a smooth, fast and seamless
          experience.

        
How does it work?

        
In a traditional web application, the browser sends a request to the
          server for a new page each time the user navigates to a different
          page. In a SPA, the browser sends a single initial request to the
          server to load the HTML, CSS and JavaScript files, and then all
          subsequent interactions with the server are done using AJAX requests.
          This means that only the data required for the current page is sent to
          the browser, instead of the full HTML code for each new page.

        
This allows the SPA to update the current page dynamically, without
          the need for a full page refresh. This leads to a much faster and
          smoother user experience, as the browser does not need to reload the
          entire page every time the user navigates to a new section of the
          application.

        
Benefits of Single Page Applications

        
    
          
  • Faster and smoother navigation for users
    • 
          
  • Improved user experience
    • 
          
  • Reduced server load
    • 
          
  • Easier to develop and maintain

          The dynamic updates are achieved through the use of JavaScript, which
          makes asynchronous requests to the server for new data. This approach
          provides a more seamless user experience compared to traditional
          multi-page web applications, where the entire page must reload to

The Problem with Single Page Apps

        

          
Single Page Apps (SPAs) are web applications that load a single
            HTML page and dynamically update its content as the user interacts
            with it.

          
However, the problem with SPAs is that they can have poor search
            engine optimization (SEO) and slow initial load times, because all
            the necessary resources, such as JavaScript, CSS, and images, are
            loaded in a single request.
